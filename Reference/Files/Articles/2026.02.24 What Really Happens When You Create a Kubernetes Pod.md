A conversation between components.

A sandbox being prepared.  
Containers being wired with networking.

Resources being tracked.

## Step 1: You Submit the Pod Manifest

Everything starts with your YAML file.

You define:

- Container image
- Ports
- Volumes
- Environment variables
- Resource limits

When you apply it, the request goes directly to the **API Server**.

Think of the API Server as the front door of Kubernetes.  
Every single action in the cluster goes through it.

The API Server:

- Validates your request
- Stores the desired state in **etcd**
- Makes the Pod object official

At this point…

The Pod exists in Kubernetes  
But it’s not running anywhere yet.

## Step 2: The Scheduler Makes a Decision

Now the **Scheduler** wakes up.

Its job?

Find the best node for your Pod.

It looks at:

- CPU & memory availability
- Taints and tolerations
- Node selectors
- Affinity rules
- Resource pressure

Then it selects a node.

**Important:**  
The Scheduler doesn’t run the Pod.  
It just says:

> _“This Pod should run on Node-2.”_

And updates the API Server.

## Step 3: Kubelet Takes Control

Now the real action begins.

Each node has a **Kubelet** agent running.

When the Pod is assigned to a node:

1. Kubelet watches the API Server
2. Sees that a Pod is assigned to _its_ node
3. Starts preparing the environment

This is where the Pod Sandbox is created.

## Step 4: Creating the Pod Sandbox

Before containers start, Kubernetes prepares the environment.

This includes:

## 🔹 Container Runtime

Docker / containerd pulls the image from the Image Registry.

If the image isn’t present locally:  
→ It fetches it first.

## Network Namespace

Every Pod gets:

- Its own IP address
- Its own network namespace
- Internal communication between containers

Containers inside the same Pod share:

- Network
- localhost

## Volumes

If volumes are defined:

- They are attached
- Mounted inside containers

Only after all this is ready…

Containers start running.

## Step 5: Pod Status Changes

Now your Pod transitions through different phases:

- **Pending** → Waiting for scheduling or image pulling
- **Running** → At least one container is running
- **Succeeded** → Completed successfully
- **Failed** → Something went wrong
- **Unknown** → Node communication lost

This status is continuously updated by the Kubelet back to the API Server.

Kubernetes is always watching.

## Step 6: Termination Begins

What happens when you delete a Pod?

kubectl delete pod my-pod

It doesn’t just disappear.

Here’s what actually happens:

1. API Server marks the Pod for deletion
2. Kubelet receives the signal
3. Kubelet sends **SIGTERM** to containers
4. Containers get time to gracefully shut down
5. After grace period → SIGKILL (if needed)

Professional-grade shutdown.

## Step 7: Cleanup Phase

Once containers stop:

- Containers are removed
- Volumes are detached
- Network interfaces are cleaned
- Resources are released

The node becomes free again.

The lifecycle is complete.

## Why This Matters (Especially for DevOps Engineers)

Understanding this flow helps you:

- Debug stuck Pods
- Fix Pending issues
- Troubleshoot CrashLoopBackOff
- Optimize scheduling
- Design better deployments
- Explain architecture confidently in interviews

When someone asks:

You won’t say:
