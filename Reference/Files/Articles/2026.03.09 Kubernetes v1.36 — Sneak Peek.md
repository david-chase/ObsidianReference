v1.36 — Sneak Peek

After covering Timbernetes (v1.35) in my last blog, I’ve been keeping a close eye on what the Kubernetes community is brewing for the next release. Kubernetes v1.36 is officially scheduled for April 22, 2026 — and while the final feature list isn’t locked in yet, we already have enough signal from the enhancements freeze to know this release is going to be a meaningful one.

Let’s break down the highlights you should be watching.

### HPA Scale-to-Zero — Finally Enabled by Default

This one’s been a long time coming. The HPAScaleToZero feature gate has been sitting in alpha since Kubernetes v1.16 — yes, you read that right. In v1.36, this feature is expected to be enabled by default, which means the <mark style="background: #BBFABBA6;">Horizontal Pod Autoscaler can finally scale your workloads down to zero replicas</mark>.

**Why This Matters**

If you’ve ever operated staging or test environments, you know the pain of paying for idle pods that sit around doing nothing. With minReplicas: 0 now viable in HPA, you can let workloads scale down completely when there’s no traffic — and scale back up when demand returns. Think of it like turning off the lights when nobody’s in the room.

This is particularly useful for intermittent workloads, batch processing pipelines, and cost-optimized clusters where every unused pod is wasted money.

### 2. Ephemeral Service Account Tokens for Image Pulls

Security-conscious teams will love this one. Traditionally, pulling images from private registries meant storing long-lived credentials as Kubernetes Secrets. The problem? Those secrets can be leaked, shared across namespaces accidentally, or sit around forever without rotation.

Kubernetes v1.36 is pushing forward the ability to use ephemeral Kubernetes Service Account (KSA) tokens for authenticating image pulls. Instead of relying on static secrets, the kubelet can now use short-lived, auto-rotating tokens that are scoped to the pod’s identity. Think of it like replacing your house key with a smart lock that generates a new code every time you need to enter — much safer.

This builds on KEP-2535 that started gaining traction in v1.33 and went beta in v1.34. By v1.36, expect this to be a mature, production-ready approach to image pull authentication.

### 3. Smarter HPA Pod Selection

Here’s a subtle but impactful one. The HPA currently gathers metrics from all pods matching a selector — including pods that might be owned by a different controller or in an unexpected state. This can throw off scaling decisions badly.

A new alpha feature in v1.36 refines this logic to ensure the <mark style="background: #BBFABBA6;">HPA only considers metrics from pods directly managed by the target workload</mark>. If you’ve ever debugged a situation where your HPA was scaling based on metrics from stale or orphaned pods, you’ll appreciate this fix. It’s like a GPS that finally stops recalculating because of cars on a different highway.

### 4. IPVS Mode in kube-proxy — Deprecated and on Its Way Out

If you’re running kube-proxy in IPVS mode, heads up — IPVS mode was deprecated in v1.35 and is expected to be removed in v1.36. The Kubernetes community has been gradually pushing towards nftables and eBPF-based alternatives for service proxying.

If your clusters rely on IPVS, now is the time to start planning your migration to iptables (nftables backend) or explore Cilium/eBPF-based proxying. Don’t wait until the removal hits you in a patch upgrade.

### 5. Gateway API and Ingress NGINX Retirement

This is a big ecosystem shift happening around the v1.36 timeframe. The upstream Kubernetes project is retiring Ingress NGINX — the controller that practically everyone learned Kubernetes ingress with. If you’re still relying on Ingress NGINX, it’s time to evaluate alternatives like Gateway API or third-party ingress controllers.

**Gateway API is the Future**

Gateway API offers richer routing capabilities, cross-namespace references, and a more standardized approach to traffic management. Think of it as Kubernetes finally getting a proper highway system instead of the patchwork of country roads we’ve been navigating with Ingress resources. Start experimenting with Gateway API now if you haven’t already.

### 6. Improved Windows Container Support and Containerd Changes

Kubernetes v1.36 continues to improve Windows container support, making it easier to run mixed Linux/Windows workloads in the same cluster. If your organization runs .NET applications alongside microservices, this is great news.

**Containerd Heads-Up**

v1.36 is expected to be the last Kubernetes release that supports older versions of containerd. If you’re still running containerd 1.6.x, it’s time to upgrade. The Kubernetes project is aligning with containerd 2.x, so make sure your container runtime is up to date before the next release catches you off guard.

### Wrapping Up

Kubernetes v1.36 might not have dropped yet, but the roadmap is packed with meaningful changes. From HPA finally learning to scale to zero, to ephemeral tokens tightening image pull security, to the retirement of Ingress NGINX — this release is all about maturing the platform and cleaning up technical debt that’s been lingering for years.

The bottom line: Start testing these features in your staging clusters now. When v1.36 officially lands on April 22, 2026, you’ll be ready to upgrade without breaking a sweat.

Until next time, may your pods never enter CrashLoopBackOff and may your YAML indentation always be correct on the first try. (Let’s be honest, it never is.)
